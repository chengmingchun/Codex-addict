它真正解决的，不是“再做一个聊天框”
CLI Agent 已经能执行任务，但项目上下文、团队规范、使用过程和结果沉淀仍然需要一个统一入口。
01 · Context
项目上下文不再反复解释
以项目目录为中心保留会话、文件树和任务历史。切回项目时，可以沿着同一条上下文继续推进，而不是从零开始。
02 · Workflow
把好用的 Prompt 变成 Skills
探索步骤、输出规范、排查方式不再依赖个人记忆，而是以 Markdown Skill 的形式成为团队可复用资产。
03 · Delivery
输出能在内网真正落地
会话支持离线 Markdown 展示，报告只使用本地 HTML/CSS，不依赖 CDN、在线字体或在线图表渲染服务。
从“命令行工具”到“持续工作流”
Intra Codex 不改变底层 Agent，只改变团队使用 Agent 的组织方式。
以前：每次都是一次性调用
—
临时打开命令行，重新说明项目背景。
—
好的 Prompt 和排查经验散落在聊天记录里。
—
输出难以区分系统状态、Agent 回复和 stderr。
—
结果很难直接沉淀到内网 Wiki。
现在：围绕项目持续推进
✓
打开项目后自动创建会话，立即开始任务。
✓
通过 Skills 固化探索和报告输出规范。
✓
系统提示、Agent 回复和 CLI 输出清晰分层。
✓
离线 HTML/CSS 结果可以直接复制到 Wiki。
推荐使用流程
默认路径就是最常用路径：先建立项目理解，再生成可交付结果。
1
打开项目
选择本地代码目录，读取项目文件树。
2
自动建会话
系统直接创建新 Session，无需额外操作。
3
选择 Agent
从 providers.yaml 中选择合适的 CLI Provider。
4
加载 Skills
默认使用探索项目和 HTML 探索报告。
5
选择上下文
优先勾选 README、配置、入口与核心 service。
6
运行任务
输入目标，使用快捷键发送并查看实时输出。
默认内置的两个 Skills
不是为了堆数量，而是覆盖最通用、最容易形成闭环的两个动作。
skills/explore.md
探索项目
要求 Agent 先阅读 README、配置、入口、核心 service 和数据模型，再输出项目定位、模块地图、关键链路、风险点与下一步建议。强调事实与推测分离。
skills/html-report.md
HTML 探索报告
把探索结果整理成可嵌入内网 Wiki 的纯 HTML/CSS 页面。不允许 JavaScript，不允许 CDN，不依赖 PlantUML server 或 Mermaid CDN。
架构很简单，边界也很清楚
它是工作台和编排层，不是模型层。
