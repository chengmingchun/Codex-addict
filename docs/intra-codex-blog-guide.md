# 把零散 CLI Agent 组织成一个工作台：Intra Codex 使用说明

> 这不是一个新的大模型，也不是要替代现有的 `opencode`、`cac` 或 `codeagent`。Intra Codex 更像是一个桌面工作台：把你已经能在内网跑起来的 CLI Agent，组织成项目、会话、Skills、文件上下文和运行记录。

很多团队在接入 AI 编程工具时，最先遇到的不是“模型能不能写代码”，而是这些更工程化的问题：

- 每个人都在命令行里零散运行 Agent，过程不可复用。
- 一个项目聊着聊着上下文断了，下一次又要重新解释。
- Skills、规范、排查流程散落在文档里，使用时要人工复制。
- 内网环境不方便安装一堆前端依赖，也不方便访问外网服务。
- CLI 输出很长，缺少统一的会话视图和历史记录。

Intra Codex 解决的就是这些问题。

---

## 一句话介绍

**Intra Codex 是一个面向内网研发场景的 Codex-like Agent Workbench。**

它把本地或内网可用的 CLI Agent 包装成一个桌面应用，让用户可以围绕项目目录持续对话、选择上下文文件、加载 Skills，并实时查看 Agent 的 stdout/stderr 输出。

---

## 它适合什么场景？

### 1. 探索陌生项目

打开一个代码目录后，可以使用默认的“探索项目”Skill，让 Agent 先阅读 README、配置文件、入口文件、核心 service、数据模型和脚本，再输出结构化理解。

适合：

- 接手历史项目
- 快速梳理模块边界
- 面试/晋升材料整理
- 比赛题目或内网题库项目预分析

### 2. 生成可沉淀的 HTML 探索报告

默认的“HTML 探索报告”Skill 会要求 Agent 输出一个可离线打开、适合贴进 Wiki 的 HTML 报告。

这个报告不依赖 CDN、在线字体、在线图片、PlantUML server 或 Mermaid CDN。也就是说，它适合内网环境沉淀。

### 3. 统一管理多个 CLI Agent

你可以在 `config/providers.yaml` 里配置不同 Provider，例如：

- `opencode`
- `cac`
- `codeagent`
- 公司内部包装过的 Agent 命令

每个 Provider 可以配置命令、参数、并发、输入方式和环境变量。

### 4. 把文件树变成上下文入口

右侧 Project Files 可以勾选文件作为上下文。发送任务时，选中的文件路径会随请求传给后端，后续可以由 runtime 进行内容打包和上下文注入。

## 基本使用流程

### 第一步：启动应用

开发环境：

```bash
npm install
npm run dev
```

内网不方便安装依赖时，推荐使用 GitHub Actions 产出的 Windows exe。

仓库里已经提供 `Build Windows EXE` 工作流，运行后会上传：

```text
intra-codex-windows-exe
```

下载后直接在 Windows 上运行即可。

---

### 第二步：打开项目

点击左侧 **打开项目**，选择一个代码目录。

应用会自动完成三件事：

1. 读取项目文件树。
2. 创建一个新会话。
3. 切换到这个新会话。

这样用户不需要先手动创建 Session，打开项目后就能直接输入任务。

---

### 第三步：选择 Agent

右侧 Agent 面板会列出 `providers.yaml` 中配置的 Provider。

可以根据任务选择不同 Agent：

- 需要代码搜索：选更擅长 repo-search 的 Agent。
- 需要代码修改：选具备 code-edit 能力的 Agent。
- 需要公司内网封装：选 `cac` 或内部 wrapper。

---

### 第四步：选择 Skills

默认内置两个 Skills：

1. **探索项目**
   - 用于理解陌生项目。
   - 强调先探索、再结论、再建议。

2. **HTML 探索报告**
   - 用于把探索结果整理成可贴进 Wiki 的离线 HTML。
   - 要求纯 HTML/CSS、无外部资源、无在线渲染引擎。

这两个 Skill 的组合适合最常见的使用路径：

> 先探索项目，再沉淀成报告。

---

### 第五步：选择上下文文件

右侧 Project Files 支持勾选文件。

建议优先选择：

- README / 使用说明
- package.json / Cargo.toml / pom.xml
- 入口文件
- 路由文件
- 核心 service
- 数据模型
- 配置文件
- 关键脚本

不要一上来全选。上下文应该是“精准输入”，不是“把整个项目塞给模型”。

---

### 第六步：发送任务

输入框支持快捷键：

```text
Ctrl + Enter 发送
Enter 换行
```

macOS 下也支持：

```text
Command + Enter 发送
```

## 推荐 Prompt 模板

### 项目探索

```text
请基于我选择的上下文文件，先探索这个项目的整体结构。
不要急着修改代码。
请输出：项目定位、模块地图、核心链路、关键文件、风险点和下一步建议。
```

### 生成 Wiki 报告

```text
请把刚才的探索结果整理成一个适合粘贴到 Wiki 的 HTML 介绍页。
要求：只使用 HTML 和 CSS，不要使用任何 JS 脚本，不要依赖任何外部资源。
请用 div 结构组织内容，并用 CSS 实现卡片、流光、悬浮、时间线等动效。
```

### 排查问题

```text
请先基于上下文文件判断这个问题可能发生在哪条链路。
输出排查路径、关键日志点、可能原因、验证方式和修复建议。
```

## 系统提示、Agent 回复和 CLI 输出的区别

会话里会出现三类消息：

- **系统提示**：应用自身的运行状态，例如开始执行、退出码、错误信息。
- **Agent 回复**：CLI Agent 的主要输出。
- **CLI 输出**：stderr 或工具侧输出。

系统提示已经单独高亮，避免被误认为是 Agent 的业务回复。

---

## Markdown 渲染说明

会话输出支持本地 Markdown 渲染，包括：

- 标题
- 段落
- 列表
- 引用
- 表格
- 代码块
- 行内代码
- 粗体和斜体

注意：PlantUML 和 Mermaid 不会调用在线渲染引擎。它们只会作为源码代码块展示，保证内网断网环境可用。

---

## Wiki 嵌入建议

如果要把介绍页嵌入公司 Wiki，建议遵守这些约束：

1. 不使用 `<script>`。
2. 不引用 CDN。
3. 不加载在线字体。
4. 不加载在线图片。
5. 动效只使用 CSS。
6. 内容结构尽量用 `div`、`section`、`span`、`a` 等基础标签。
7. 搜索需求优先交给浏览器 Ctrl+F 或 Wiki 自带搜索。

这也是为什么 HTML 探索报告 Skill 会强调“纯 HTML/CSS”和“离线可用”。

## 这个工具的边界

Intra Codex 本身不提供模型能力。

它做的是：

- 管理项目
- 管理会话
- 管理 Agent Provider
- 管理 Skills
- 管理上下文入口
- 管理 CLI 运行过程
- 管理本地输出展示

真正执行任务的，仍然是你配置的 CLI Agent。

所以它的价值不是“又造了一个 Agent”，而是把 Agent 使用方式工程化。

---

## 最推荐的一条工作流

```text
打开项目
  ↓
自动创建会话
  ↓
选择探索项目 + HTML 探索报告 Skills
  ↓
勾选 README、配置、入口、核心 service
  ↓
让 Agent 先探索项目
  ↓
再让 Agent 输出 Wiki HTML 介绍页
  ↓
复制到 Wiki 沉淀
```

这条链路非常适合内网研发、项目交接、技术预研、比赛题目分析和个人作品集展示。

---

## 最后

Intra Codex 的目标不是炫技，而是把 AI 编程工具真正变成可复用、可沉淀、可交付的研发工作流。

如果 CLI Agent 是发动机，那么 Intra Codex 就是驾驶舱。
